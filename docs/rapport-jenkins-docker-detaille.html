1. Objectif du travail

L'objectif etait de mettre en place Jenkins pour automatiser le cycle CI/CD du projet fil rouge. Le projet contient un frontend React/Vite, une API Express, une connexion MongoDB et des scripts npm. Jenkins doit pouvoir recuperer le code, installer les dependances, verifier le projet, construire le frontend, archiver le resultat et preparer une image Docker.

Docker a ete utilise pour installer Jenkins rapidement sans modifier l'installation Windows. Jenkins tourne donc dans un conteneur, avec ses donnees conservees dans un volume Docker.

4. Verification de Docker Desktop

Avant d'installer Jenkins, il fallait verifier que Docker Desktop etait lance et que le moteur Docker repondait. Sans le moteur Docker, la commande docker run ne peut pas creer le conteneur Jenkins. 

docker info

A quoi ca sert : afficher l'etat du client Docker et du serveur Docker.

Exemple de resultat utile :

Server:
 Containers: 3
 Server Version: 29.4.2
 Operating System: Docker Desktop
 OSType: linux
 Architecture: x86_64
La verification a confirme que Docker Desktop etait actif. Jenkins pouvait donc etre installe avec Docker.

5. Verification d'une ancienne installation Jenkins

Avant de creer un nouveau conteneur, j'ai verifie s'il existait deja un conteneur, un volume ou un reseau Jenkins. Cela evite d'ecraser une installation existante. 

docker ps -a --filter name=jenkins --format "{{.Names}}\t{{.Status}}\t{{.Ports}}"
docker volume ls --filter name=jenkins_home --format "{{.Name}}"
docker network ls --filter name=jenkins --format "{{.Name}}"
Commande A quoi ca sert Exemple
docker ps -a --filter name=jenkins Voir si un conteneur Jenkins existe deja, meme s'il est arrete. Si rien ne s'affiche, aucun conteneur Jenkins n'existe.
docker volume ls --filter name=jenkins_home Voir si un volume Jenkins existe deja. Si jenkins_home existe, les donnees Jenkins sont deja presentes.
docker network ls --filter name=jenkins Voir si le reseau Jenkins existe deja. Si rien ne s'affiche, on peut creer le reseau.

Dans ce cas, aucune ancienne installation Jenkins n'a ete detectee. L'installation pouvait donc commencer proprement.

6. Creation du reseau Docker Jenkins

docker network create jenkins

A quoi ca sert : creer un reseau Docker nomme jenkins.

Meme si Jenkins peut fonctionner avec le reseau Docker par defaut, creer un reseau dedie rend l'installation plus propre. Plus tard, si on ajoute un agent Jenkins, un conteneur Docker-in-Docker ou un autre service, on pourra les connecter au meme reseau.

Exemple concret : un agent Jenkins pourrait rejoindre ce reseau pour communiquer avec le serveur Jenkins.

docker run --network jenkins mon-agent-jenkins

7. Creation du volume Jenkins

docker volume create jenkins_home

A quoi ca sert : creer un stockage persistant pour Jenkins.

Jenkins stocke ses donnees dans /var/jenkins_home a l'interieur du conteneur. Si ces donnees restent uniquement dans le conteneur, elles peuvent etre perdues en cas de suppression du conteneur. Le volume jenkins_home evite ce probleme.

Sans volume Avec volume
Suppression du conteneur = risque de perdre la configuration Jenkins. Suppression du conteneur = les donnees restent dans jenkins_home.

Exemple : les plugins installes, les jobs crees et les utilisateurs Jenkins sont gardes dans ce volume.

8. Installation et lancement de Jenkins

docker run --name jenkins -d `
  --restart=on-failure `
  --network jenkins `
  -p 8080:8080 -p 50000:50000 `
  -v jenkins_home:/var/jenkins_home `
  jenkins/jenkins:lts-jdk21

Cette commande est le coeur de l'installation Jenkins.

Partie de la commande A quoi ca sert Exemple ou effet
docker run Creer et demarrer un conteneur depuis une image Docker. Docker cree le conteneur Jenkins.
--name jenkins Donner un nom simple au conteneur. On peut ensuite utiliser docker logs jenkins.
-d Lancer le conteneur en arriere-plan. Le terminal reste disponible pendant que Jenkins tourne.
--restart=on-failure Relancer Jenkins automatiquement si le conteneur s'arrete a cause d'une erreur. Utile pour garder Jenkins disponible apres un crash.
--network jenkins Connecter le conteneur au reseau Docker cree plus haut. Jenkins appartient au reseau jenkins.
-p 8080:8080 Exposer l'interface web Jenkins. Acces navigateur : http://localhost:8080.
-p 50000:50000 Exposer le port utilise par certains agents Jenkins. Utile si Jenkins pilote des agents externes.
-v jenkins_home:/var/jenkins_home Connecter le volume persistant au dossier Jenkins. Les donnees Jenkins restent sauvegardees.
jenkins/jenkins:lts-jdk21 Image Jenkins officielle LTS avec Java 21. Docker la telecharge si elle n'existe pas localement.

Resultat : Docker a telecharge l'image Jenkins puis a demarre le conteneur.

A quoi ca sert : executer une commande a l'interieur du conteneur Jenkins pour lire le fichier du mot de passe initial.

Exemple d'utilisation : copier ce mot de passe, ouvrir Jenkins dans le navigateur, puis le coller dans l'ecran "Unlock Jenkins".

15. Creation du Jenkinsfile

J'ai ajoute un fichier Jenkinsfile a la racine du projet. Ce fichier decrit tout ce que Jenkins doit faire. C'est ce qu'on appelle un pipeline as code.

A quoi ca sert : le pipeline devient versionne avec le projet. Si le projet est clone ailleurs, Jenkins sait toujours comment le construire.

pipeline {
    agent any
    stages {
        stage("Checkout") {
            steps {
                checkout scm
            }
        }
        stage("Install dependencies") {
            steps {
                script {
                    runCommand("npm ci")
                }
            }
        }
        stage("Build frontend") {
            steps {
                script {
                    runCommand("npm run build")
                }
            }
        }
    }
}

Exemple : quand on clique sur Build Now, Jenkins lit ce fichier et execute les etapes dans l'ordre.

16. Detail des stages Jenkins ajoutes

Install dependencies Installer les dependances exactes du projet depuis package-lock.json. npm ci

17. Pourquoi j'ai utilise npm ci dans Jenkins

npm ci est plus adapte a la CI que npm install. Il installe les dependances a partir de package-lock.json et donne un environnement plus reproductible.

Commande Utilisation Exemple
npm install Pratique en developpement, peut modifier le lockfile. Ajouter une dependance localement.
npm ci Pratique en CI, installation propre et reproductible. Utilise dans Jenkins.

19. Adaptation de Express pour servir dist en production

Le projet avait deja une API Express. J'ai adapte app.js pour que, en production, Express serve aussi les fichiers generes par Vite dans dist/. 

if (process.env.NODE_ENV === "production") {
    app.use(express.static(distPath));
    app.get(/^(?!\/api).*/, (_req, res) => {
        return res.sendFile(path.join(distPath, "index.html"));
    });
}
Element A quoi ca sert Exemple
express.static(distPath) Servir les fichiers HTML, CSS et JS generes. /assets/index.js est envoye au navigateur.
app.get(/^(?!\/api).*/) Rediriger les routes React vers index.html. /portfolio affiche l'application React.
NODE_ENV=production Activer le comportement de production. Utilise dans le Dockerfile runtime.

20. Creation du Dockerfile du projet

J'ai ajoute un Dockerfile multi-stage. Cela permet de construire le frontend dans une premiere etape, puis de garder une image finale plus propre pour l'execution. 

FROM node:22-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:22-alpine AS runtime
WORKDIR /app
ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev && npm cache clean --force
COPY app.js ./
COPY config ./config
COPY controllers ./controllers
COPY lib ./lib
COPY middleware ./middleware
COPY models ./models
COPY public ./public
COPY routes ./routes
COPY --from=build /app/dist ./dist
EXPOSE 5000
CMD ["node", "app.js"]
Ligne A quoi ca sert Exemple
FROM node:22-alpine AS build Utiliser Node.js pour construire le projet. Vite a besoin de Node pour generer dist/.
RUN npm ci Installer toutes les dependances, y compris les dependances de build. Installer Vite et React.
RUN npm run build Construire le frontend. Creer dist/index.html.
FROM node:22-alpine AS runtime Creer l'image finale d'execution. Elle contient l'API et le build final.
npm ci --omit=dev Installer seulement les dependances necessaires en production. Express, mongoose, dotenv, cors.
COPY --from=build /app/dist ./dist Copier le build frontend dans l'image finale. Express peut servir dist/.
EXPOSE 5000 Documenter le port utilise par l'application. L'application Express ecoute sur 5000.
CMD ["node", "app.js"] Definir la commande lancee au demarrage du conteneur. Demarrer l'API Express.

26. Gestion des credentials Jenkins

Les credentials permettent de garder les secrets hors du code source. Il ne faut pas ecrire les mots de passe dans le Jenkinsfile.

31. Problemes possibles et solutions

Probleme Cause possible Solution
Cannot connect to Docker daemon Docker Desktop n'est pas lance. Ouvrir Docker Desktop puis relancer docker info.
Port 8080 deja utilise Une autre application utilise le port. Lancer Jenkins avec -p 8081:8080.
Mot de passe initial introuvable Jenkins est deja configure ou le fichier a ete supprime. Verifier les logs ou utiliser le compte admin cree.
Pipeline echoue sur npm ci Node.js absent sur l'agent Jenkins ou lockfile incoherent. Installer/configurer NodeJS dans Jenkins ou corriger package-lock.json.
Stage Docker echoue dans Jenkins Docker n'est pas accessible depuis l'agent Jenkins. Installer Docker sur l'agent ou configurer un agent dedie.
